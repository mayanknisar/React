Explain async/await and common pitfalls in high-throughput enterprise applications
Core★ High priority
▼
Aasync/await wraps the Task Parallel Library allowing non-blocking I/O without manual thread management. The compiler rewrites the method as a state machine. Key pitfalls in enterprise apps: (1) async void — swallows exceptions; always use async Task. (2) .Result / .Wait() deadlocks in ASP.NET contexts — always await. (3) ConfigureAwait(false) in library code to avoid capturing the synchronisation context unnecessarily. (4) Not cancelling long-running operations — always accept and honour CancellationToken. (5) Unbounded parallelism — use SemaphoreSlim or Channel<T> to throttle concurrent work rather than firing thousands of Tasks.
// Good pattern for cancellable, throttled bulk work
public async Task ProcessBatchAsync(
    IEnumerable<int> ids, CancellationToken ct)
{
    var sem = new SemaphoreSlim(10); // max 10 concurrent
    var tasks = ids.Select(async id => {
        await sem.WaitAsync(ct);
        try { await ProcessOneAsync(id, ct); }
        finally { sem.Release(); }
    });
    await Task.WhenAll(tasks);
}
What is the difference between IEnumerable, IQueryable, and IAsyncEnumerable?
Core
▼
AIEnumerable<T> executes in-memory — once data is pulled from the database it iterates in the application process. IQueryable<T> builds an expression tree that is translated to SQL and executed on the database server — filtering and projection happen in the DB, not in memory. Critical for EF Core: always filter before materialising with ToList(). IAsyncEnumerable<T> (C# 8+) enables async streaming — yielding items one-by-one as they arrive without buffering the full result set, ideal for large result sets or server-sent events. EF Core supports it via ToAsyncEnumerable() and await foreach.
How does Dependency Injection work in .NET 8 and what is keyed DI?
Core★ High priority
▼
AThe built-in DI container in ASP.NET Core registers services in Program.cs and resolves them via constructor injection. Lifetimes: Singleton (one per process), Scoped (one per HTTP request), Transient (new per injection). Keyed DI (.NET 8) solves the problem of needing multiple implementations of the same interface — you register with a key and resolve with [FromKeyedServices] or GetRequiredKeyedService<T>.
// Register two INotificationService implementations
builder.Services.AddKeyedScoped<INotificationService, EmailService>("email");
builder.Services.AddKeyedScoped<INotificationService, SmsService>("sms");

// Resolve by key in constructor
public class OrderService(
    [FromKeyedServices("email")] INotificationService emailSvc,
    [FromKeyedServices("sms")]   INotificationService smsSvc) { }
Explain the Blazor render modes in .NET 8 and when you'd choose each
Blazor★ High priority
▼
A.NET 8 introduced per-component render modes under the Blazor United model: Static SSR — HTML rendered server-side, no interactivity, best for SEO-heavy pages. Interactive Server — component runs on server with SignalR, low initial load, requires persistent connection, suits internal enterprise dashboards. Interactive WebAssembly — runs in browser after download, no server connection needed post-load, suits offline or high-interactivity scenarios. Interactive Auto — starts as Server while WASM downloads, then switches to WASM silently. For this enterprise JD, Interactive Server is the practical default for LOB apps — it avoids large WASM downloads on corporate networks and keeps sensitive business logic server-side.
JD angle: The JD specifically calls out "Blazor Server" — know the trade-offs: SignalR dependency, server memory per user, but excellent for authenticated enterprise intranet apps.
How do you handle state management in a complex Blazor enterprise application?
Blazor★ High priority
▼

Scoped service state containers for cross-component state within a user session — a plain C# class with an event Action OnChange, injected as Scoped DI, components subscribe in OnInitialized and unsubscribe in Dispose. How do you optimise Blazor rendering performance for data-heavy enterprise screens?
Blazor
▼
AKey techniques I apply: (1) Virtualise long lists with <Virtualize> — only renders visible rows, essential for ERP grids with thousands of records. (2) Override ShouldRender() on stable child components to skip unnecessary diffs. (3) Use @key on list items so the diffing algorithm tracks identity correctly on reorder/insert. (4) Break large pages into smaller focused components — only the changed subtree re-renders. (5) Prefer EventCallback over Action — it auto-calls StateHasChanged on the right component, avoiding double renders. (6) Debounce rapid input events (search-as-you-type) with a timer before triggering API calls.

How do you optimise EF Core queries to avoid N+1 and over-fetching in enterprise apps?
EF Core★ High priority
▼
AN+1 is the biggest EF Core trap — loading a list then accessing a navigation property per row triggers a query per row. Fix with eager loading (Include/ThenInclude) or explicit loading where needed. Over-fetching (pulling full entities when you need two columns) wastes memory and I/O — use projection: Select(x => new CustomerDto { Id = x.Id, Name = x.Name }). Other optimisations: AsNoTracking() for read-only queries (no change tracker overhead). Split queries for large Include chains to avoid cartesian explosion. Compiled queries for hot paths. EnableSensitiveDataLogging + logging to catch unexpected query patterns in development. For bulk operations, skip EF entirely and use Dapper or EF Core's ExecuteUpdate/ExecuteDelete (EF 7+) to avoid loading entities just to update them.
// ExecuteUpdate — no entity load required (EF 7+)
await ctx.Orders
    .Where(o => o.Status == OrderStatus.Pending && o.CreatedAt < cutoff)
    .ExecuteUpdateAsync(s => s.SetProperty(o => o.Status, OrderStatus.Expired));
Explain your approach to database migrations in a CI/CD pipeline
SQL / DevOps
▼
AI treat migrations as code — they live in the repository alongside application code. In the CI pipeline I run dotnet ef migrations script --idempotent to generate a SQL script (not Database.Migrate() at runtime — too risky in production). In the CD pipeline, that SQL script is applied to the target environment by the pipeline service account with limited permissions. For ERP integrations I follow expand-contract pattern: Phase 1 (expand) adds new columns as nullable. Phase 2 deploys app code that reads/writes the new column. Phase 3 (contract) makes the column non-nullable or removes the old one. This ensures zero-downtime deployments with rolling deploys.
How do you implement the Repository pattern with EF Core — and should you?
EF Core
▼
ADbContext is already a Unit of Work and DbSet<T> is already a repository — wrapping them in another repository layer often adds indirection without value, especially since IQueryable composability is lost. My view: use the repository pattern when you need to abstract away the data store for testability or when genuinely switching providers. In practice, I inject DbContext directly in application services and use Specification pattern for reusable query logic. For testing, I use an in-memory SQLite database or mock DbContext with EF Core's UseInMemoryDatabase — which is sufficient for most unit tests without a full repository abstraction.
Stand-out answer: Showing you understand the trade-offs rather than blindly applying the pattern demonstrates senior-level thinking that interviewers value.

How have you used Azure Service Bus in enterprise application integration?
Azure
▼
AService Bus decouples services and handles reliability at the messaging layer. Key patterns I've used: Topics/subscriptions for pub-sub — an OrderPlaced event published once, consumed independently by Fulfilment, Billing, and Notification services. Explain Clean Architecture and how you apply it in a .NET enterprise solution
Architecture★ High priority
▼
AClean Architecture organises code into concentric layers where dependencies only point inward: Domain (entities, value objects, domain events — zero external dependencies) → Application (use cases, interfaces, DTOs, MediatR handlers — depends only on Domain) → Infrastructure (EF Core, Azure SDKs, HTTP clients — implements interfaces defined in Application) → Presentation (Blazor components, API controllers — depends on Application). The key rule: Infrastructure depends on Application, never the reverse. This means you can swap SQL Server for Cosmos DB or replace Blazor with an API without touching business logic. In practice I use MediatR for the Application layer — each use case is a Command or Query handler, keeping controllers/components thin.
// Application layer — no infrastructure knowledge
public record CreateOrderCommand(Guid CustomerId, List<OrderLine> Lines)
    : IRequest<OrderId>;

public class CreateOrderHandler(IOrderRepository repo, IEventBus bus)
    : IRequestHandler<CreateOrderCommand, OrderId>
{
    public async Task<OrderId> Handle(CreateOrderCommand cmd, CancellationToken ct)
    {
        var order = Order.Create(cmd.CustomerId, cmd.Lines);
        await repo.AddAsync(order, ct);
        await bus.PublishAsync(new OrderCreatedEvent(order.Id), ct);
        return order.Id;
    }
}
What is CQRS and when would you apply it in this type of enterprise context?
Architecture★ High priority
▼
ACQRS separates read (Query) from write (Command) models. Commands mutate state and return only an ID or void — they go through the full domain model with validation and business rules. Queries return DTOs optimised for the UI — they can bypass the domain model entirely and hit a read-optimised store or use raw SQL/Dapper for performance. For ERP-adjacent applications in this JD: write side uses EF Core + domain events, read side uses Dapper against read-replica Azure SQL for reporting and dashboard queries. I use MediatR to dispatch both — the Presentation layer doesn't know whether it's sending a command or query. I don't apply CQRS everywhere — simple CRUD screens don't benefit; it's justified when read/write load profiles differ significantly or when the write model is complex.

Describe your testing strategy for a .NET enterprise application end-to-end
Testing★ High priority
▼
AI follow the test pyramid. Unit tests (NUnit + FluentAssertions + Moq/NSubstitute) for domain logic and application handlers in isolation — fast, no I/O, run on every commit. Integration tests (WebApplicationFactory + testcontainers for real SQL Server in Docker) for API endpoints and EF Core queries against a real database schema — catches mapping and query issues unit tests miss. How do you design RESTful APIs for enterprise system integration (ERP, CRM)?
APIs
▼
AFor enterprise integration I follow these practices: versioned APIs (/api/v1/) with deprecation notices before breaking changes — critical when D365 or CRM consume your endpoints. Problem Details (RFC 7807) for consistent error responses across all endpoints. Pagination with cursor-based tokens for large datasets (offset pagination breaks under concurrent inserts). Idempotency keys on mutation endpoints so D365 retries don't create duplicates. OpenAPI/Swagger spec-first or spec-generated with Swashbuckle, shared as a package with consumer teams. Rate limiting middleware (ASP.NET Core RateLimiter in .NET 7+). For outbound calls to D365/CRM I use Polly for retry with exponential back-off and circuit breaker — D365 has rate limits and transient failures are common.
