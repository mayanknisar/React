What is PL/SQL?
Language basics, block structure, use cases
▼

PL/SQL (Procedural Language / Structured Query Language) is Oracle's procedural extension to SQL. It adds programming constructs — variables, loops, conditions, and exception handling — to the declarative SQL language, allowing complex business logic to run directly in the database.

Every PL/SQL program is a block made of up to four sections:

DECLARE
  -- optional: variables, cursors, exceptions
  v_name  VARCHAR2(50) := 'Oracle';
  v_count NUMBER;
BEGIN
  -- mandatory: executable statements
  v_count := 10;
  DBMS_OUTPUT.PUT_LINE('Hello, ' || v_name);
EXCEPTION
  -- optional: error handling
  WHEN OTHERS THEN
    DBMS_OUTPUT.PUT_LINE('Error: ' || SQLERRM);
END;
/
Key advantage: Reduces network round-trips by bundling multiple SQL calls into one server-side execution unit.
QWhat is PL/SQL and how does it differ from SQL?
ASQL is a declarative language used to query and manipulate data — you tell the database what you want. PL/SQL is a procedural extension that lets you express how to do it using variables, conditionals, loops, and exception handlers. It integrates seamlessly with SQL and runs entirely in the Oracle database engine, reducing network overhead and improving performance for complex operations.
QWhat are the sections of a PL/SQL block and which are mandatory?
AA PL/SQL block has four sections: DECLARE (optional — variables, constants, cursors), BEGIN (mandatory — executable statements), EXCEPTION (optional — error handling), and END (mandatory — closes the block). Only BEGIN...END is required; the rest are optional.
Variables, Data Types & Constants
Declaring, initializing, %TYPE, %ROWTYPE
▼

Variables are declared in the DECLARE section. PL/SQL supports scalar types, composite types, and anchor types that bind to database column definitions.

DECLARE
  v_salary    NUMBER(10,2) := 5000.00;
  v_name      VARCHAR2(100);
  v_hired     DATE         := SYSDATE;
  c_max_sal   CONSTANT NUMBER := 100000;

  -- %TYPE anchors to a column's data type
  v_emp_sal   employees.salary%TYPE;

  -- %ROWTYPE holds an entire row
  v_emp_row   employees%ROWTYPE;
BEGIN
  SELECT salary INTO v_emp_sal
  FROM   employees
  WHERE  employee_id = 101;

  SELECT * INTO v_emp_row
  FROM   employees
  WHERE  employee_id = 101;

  DBMS_OUTPUT.PUT_LINE(v_emp_row.last_name);
END;
/
Best practice: Always use %TYPE and %ROWTYPE instead of hardcoding data types. If the column definition changes, your PL/SQL automatically adapts at compile time.
QWhat is the difference between %TYPE and %ROWTYPE?
A%TYPE anchors a variable to the data type of a single column (e.g., employees.salary%TYPE gives you a NUMBER matching that column's precision). %ROWTYPE anchors a variable to the full structure of a table row or cursor result — it creates a record with one field per column. Use %TYPE for individual column values and %ROWTYPE when you need to hold or pass an entire row.
QWhat is the difference between VARCHAR2 and CHAR in PL/SQL?
ACHAR is a fixed-length type — it always uses the full declared length, padding with spaces. VARCHAR2 is variable-length — it only stores the actual characters. For most use cases VARCHAR2 is preferred because it avoids trailing space issues in comparisons. CHAR is only appropriate when values are genuinely fixed-length (e.g., 2-char country codes).
IF / ELSIF / ELSE
Conditional branching
▼ 
DECLARE
  v_score NUMBER := 75;
  v_grade VARCHAR2(2);
BEGIN
  IF v_score >= 90 THEN
    v_grade := 'A';
  ELSIF v_score >= 75 THEN
    v_grade := 'B';
  ELSIF v_score >= 60 THEN
    v_grade := 'C';
  ELSE
    v_grade := 'F';
  END IF;
  DBMS_OUTPUT.PUT_LINE('Grade: ' || v_grade);
END;
/
QWhat happens if a NULL value is compared in an IF condition?
AAny comparison involving NULL evaluates to NULL (not TRUE or FALSE) in PL/SQL. An IF condition that evaluates to NULL is treated as FALSE and the ELSE branch executes. Always use IS NULL or IS NOT NULL to explicitly test for NULL values rather than = NULL or != NULL.
Loops — LOOP, WHILE, FOR
Basic, while, numeric FOR, cursor FOR
▼ 
-- Basic LOOP (must use EXIT)
DECLARE i NUMBER := 1; BEGIN
  LOOP
    DBMS_OUTPUT.PUT_LINE(i);
    i := i + 1;
    EXIT WHEN i > 5;
  END LOOP;
END;
/

-- WHILE LOOP
WHILE i <= 10 LOOP
  i := i + 1;
END LOOP;

-- Numeric FOR LOOP (i is implicit, read-only)
FOR i IN 1..10 LOOP
  DBMS_OUTPUT.PUT_LINE(i);
END LOOP;

-- Reverse FOR
FOR i IN REVERSE 1..5 LOOP
  DBMS_OUTPUT.PUT_LINE(i); -- 5,4,3,2,1
END LOOP;
QCan you modify the loop counter variable in a FOR loop?
ANo. In a numeric FOR loop, the loop counter (e.g., i) is implicitly declared and is read-only within the loop body. Any attempt to assign a value to it results in a compile-time error. If you need a modifiable counter, use a WHILE loop or a LOOP...EXIT WHEN construct with an explicitly declared variable.
Implicit vs Explicit Cursors
SQL%ROWCOUNT, OPEN/FETCH/CLOSE
▼

A cursor is a pointer to a private SQL work area that holds the result of a query. Implicit cursors are created automatically for single-row SELECT INTO and DML statements. Explicit cursors are declared and controlled manually.

-- Implicit cursor attributes
BEGIN
  UPDATE employees SET salary = salary * 1.1
  WHERE  department_id = 10;
  DBMS_OUTPUT.PUT_LINE(SQL%ROWCOUNT || ' rows updated');
END;

-- Explicit cursor
DECLARE
  CURSOR c_emp IS
    SELECT employee_id, last_name, salary
    FROM   employees
    WHERE  department_id = 20;
  v_emp c_emp%ROWTYPE;
BEGIN
  OPEN  c_emp;
  LOOP
    FETCH c_emp INTO v_emp;
    EXIT WHEN c_emp%NOTFOUND;
    DBMS_OUTPUT.PUT_LINE(v_emp.last_name);
  END LOOP;
  CLOSE c_emp;
END;
/
Shortcut: Cursor FOR loop handles OPEN, FETCH, and CLOSE automatically — prefer it when you don't need manual control.
FOR rec IN (
  SELECT last_name, salary FROM employees WHERE dept_id = 10
) LOOP
  DBMS_OUTPUT.PUT_LINE(rec.last_name || ' - ' || rec.salary);
END LOOP;
QWhat are cursor attributes and what do they return?
A%FOUND — TRUE if the last FETCH returned a row. %NOTFOUND — TRUE if the last FETCH returned no row. %ROWCOUNT — number of rows fetched so far. %ISOPEN — TRUE if the cursor is currently open. For implicit cursors, these apply to the most recent SQL statement. For explicit cursors, they apply to the named cursor. Always check %NOTFOUND inside an explicit cursor loop to avoid infinite iteration.
QWhat happens if you forget to close an explicit cursor?
AThe cursor remains open, consuming memory (cursor slots) in the Oracle shared pool. If enough cursors are left open, you may hit the OPEN_CURSORS limit and receive ORA-01000: maximum open cursors exceeded. Always close cursors when done — ideally in the EXCEPTION section as well so they're closed even on error.
Parameterized & REF Cursors
Dynamic queries, SYS_REFCURSOR
▼ 
-- Parameterized cursor
DECLARE
  CURSOR c_dept(p_dept_id NUMBER) IS
    SELECT last_name FROM employees
    WHERE  department_id = p_dept_id;
BEGIN
  FOR r IN c_dept(10) LOOP
    DBMS_OUTPUT.PUT_LINE(r.last_name);
  END LOOP;
END;
/

-- REF CURSOR (weak type) — return from procedure
CREATE OR REPLACE PROCEDURE get_employees(
  p_dept  IN  NUMBER,
  p_cur   OUT SYS_REFCURSOR
) AS BEGIN
  OPEN p_cur FOR
    SELECT * FROM employees WHERE department_id = p_dept;
END;
/
SYS_REFCURSOR is the standard way to return result sets from stored procedures to calling applications (Java, .NET, etc.).
QWhat is the difference between a strong and weak REF CURSOR?
AA strong REF CURSOR is declared with a specific return type (RETURN employees%ROWTYPE) — it is type-safe and allows compile-time checking. A weak REF CURSOR (SYS_REFCURSOR or declared without a RETURN clause) can hold any query result but provides no compile-time type checking. Weak cursors are more flexible and commonly used to return dynamic result sets to applications.
Built-in vs User-Defined Exceptions
NO_DATA_FOUND, TOO_MANY_ROWS, RAISE
▼ 
DECLARE
  v_sal employees.salary%TYPE;
  e_too_low EXCEPTION;          -- user-defined
BEGIN
  SELECT salary INTO v_sal
  FROM   employees
  WHERE  employee_id = 999;     -- may raise NO_DATA_FOUND

  IF v_sal < 1000 THEN
    RAISE e_too_low;             -- raise custom exception
  END IF;

EXCEPTION
  WHEN NO_DATA_FOUND THEN
    DBMS_OUTPUT.PUT_LINE('Employee not found');
  WHEN TOO_MANY_ROWS THEN
    DBMS_OUTPUT.PUT_LINE('More than one row returned');
  WHEN e_too_low THEN
    DBMS_OUTPUT.PUT_LINE('Salary below minimum');
  WHEN OTHERS THEN
    DBMS_OUTPUT.PUT_LINE('Error: ' || SQLERRM);
    RAISE;  -- re-raise to caller
END;
/
Caution: WHEN OTHERS without RAISE silently swallows errors. Always log (SQLERRM / DBMS_UTILITY.FORMAT_ERROR_BACKTRACE) and re-raise in production code.
QWhat is RAISE_APPLICATION_ERROR and when would you use it?
ARAISE_APPLICATION_ERROR lets you raise a custom error with a user-defined error number (between -20000 and -20999) and a descriptive message. It propagates back to the calling application just like a system error, making it ideal for enforcing business rules. Example: RAISE_APPLICATION_ERROR(-20001, 'Salary cannot exceed department budget'); The calling application receives an ORA-20001 error with your custom message.
QWhat is PRAGMA EXCEPTION_INIT?
APRAGMA EXCEPTION_INIT associates a user-defined exception name with a specific Oracle error number, letting you trap Oracle errors by a readable name. Example: PRAGMA EXCEPTION_INIT(e_fk_violation, -2292); Now you can WHEN e_fk_violation THEN instead of catching it in WHEN OTHERS and inspecting SQLCODE.
Stored Procedures
IN, OUT, IN OUT parameters
▼ 
CREATE OR REPLACE PROCEDURE update_salary(
  p_emp_id  IN     employees.employee_id%TYPE,
  p_pct     IN     NUMBER,
  p_new_sal    OUT  employees.salary%TYPE
) AS
BEGIN
  UPDATE employees
  SET    salary = salary * (1 + p_pct/100)
  WHERE  employee_id = p_emp_id
  RETURNING salary INTO p_new_sal;

  IF SQL%ROWCOUNT = 0 THEN
    RAISE_APPLICATION_ERROR(-20001, 'Employee not found');
  END IF;
END update_salary;
/

-- Calling the procedure
DECLARE
  v_new_sal NUMBER;
BEGIN
  update_salary(101, 10, v_new_sal);
  DBMS_OUTPUT.PUT_LINE('New salary: ' || v_new_sal);
END;
/
QWhat is the difference between IN, OUT, and IN OUT parameters?
AIN — read-only input. The procedure cannot modify it. Default mode. OUT — write-only output. Any value passed in is not accessible inside; the procedure writes a result back to the caller. IN OUT — both: the procedure reads the incoming value and can also modify it to return a result. Use IN OUT when you need to transform a value in place.
Functions vs Procedures
RETURN, calling from SQL, deterministic
▼ 
CREATE OR REPLACE FUNCTION get_annual_salary(
  p_emp_id IN employees.employee_id%TYPE
) RETURN NUMBER
DETERMINISTIC  -- same input always gives same output
AS
  v_sal NUMBER;
BEGIN
  SELECT salary * 12 INTO v_sal
  FROM   employees
  WHERE  employee_id = p_emp_id;
  RETURN v_sal;
EXCEPTION
  WHEN NO_DATA_FOUND THEN RETURN NULL;
END;
/

-- Use directly in SQL
SELECT last_name, get_annual_salary(employee_id) AS annual
FROM   employees;
DETERMINISTIC hint: Tells the optimizer it can cache function results for repeated identical inputs — useful in queries scanning large tables.
QWhat restrictions apply when calling a PL/SQL function from SQL?
AFunctions called from SQL must: (1) not modify database tables (no INSERT/UPDATE/DELETE) unless called from a DML statement that already has that privilege, (2) not use COMMIT or ROLLBACK, (3) not reference package variables that change between calls in ways that affect purity, and (4) take only IN parameters (no OUT or IN OUT). The PRAGMA RESTRICT_REFERENCES directive (older) or DETERMINISTIC keyword helps declare these constraints to the optimizer.
DML Triggers
BEFORE/AFTER, row-level, :NEW/:OLD
▼ 
CREATE OR REPLACE TRIGGER trg_audit_salary
AFTER UPDATE OF salary ON employees
FOR EACH ROW
WHEN (NEW.salary <> OLD.salary)
BEGIN
  INSERT INTO salary_audit(
    emp_id, old_sal, new_sal, changed_by, changed_on
  ) VALUES (
    :NEW.employee_id,
    :OLD.salary,
    :NEW.salary,
    USER,
    SYSDATE
  );
END;
/
:NEW — the new value after DML. :OLD — the previous value. In a BEFORE trigger you can modify :NEW to change what gets written. :OLD is read-only always.
QWhat is a mutating table error (ORA-04091) and how do you resolve it?
AORA-04091 occurs when a row-level trigger tries to query or modify the same table that fired the trigger — the table is "mutating" (mid-DML) so Oracle prevents reading potentially inconsistent data. Solutions: (1) Use a statement-level trigger instead of row-level if you don't need :NEW/:OLD values. (2) Collect values in a package-level collection during row-level trigger execution, then process them in an AFTER STATEMENT trigger. (3) In Oracle 11g+, use compound triggers which bundle row and statement sections together cleanly.
QWhat is the difference between a BEFORE and AFTER trigger?
AA BEFORE trigger fires before the DML operation completes — useful for validating or modifying :NEW values before they are written. An AFTER trigger fires after the operation — useful for audit logging or cascading actions where you need the confirmed final values. BEFORE triggers can cancel the operation by raising an exception; AFTER triggers cannot change what was written.
Package Specification & Body
Public/private, overloading, state
▼

A package groups related procedures, functions, variables, and types. The specification declares the public interface; the body contains the implementation.

-- Package Specification (public interface)
CREATE OR REPLACE PACKAGE emp_pkg AS
  g_default_dept  NUMBER := 10;     -- package variable

  PROCEDURE hire_employee(p_name VARCHAR2, p_sal NUMBER);
  FUNCTION  get_headcount(p_dept NUMBER) RETURN NUMBER;
  FUNCTION  get_headcount RETURN NUMBER; -- overloaded
END emp_pkg;
/

-- Package Body (private + implementation)
CREATE OR REPLACE PACKAGE BODY emp_pkg AS
  -- Private helper (not in spec)
  FUNCTION validate_sal(p_sal NUMBER) RETURN BOOLEAN IS
  BEGIN RETURN p_sal > 0; END;

  PROCEDURE hire_employee(p_name VARCHAR2, p_sal NUMBER) IS
  BEGIN
    IF NOT validate_sal(p_sal) THEN
      RAISE_APPLICATION_ERROR(-20001, 'Invalid salary');
    END IF;
    -- insert logic here
  END;

  FUNCTION get_headcount(p_dept NUMBER) RETURN NUMBER IS
    v_cnt NUMBER;
  BEGIN
    SELECT COUNT(*) INTO v_cnt
    FROM   employees WHERE department_id = p_dept;
    RETURN v_cnt;
  END;

  FUNCTION get_headcount RETURN NUMBER IS
  BEGIN RETURN get_headcount(g_default_dept); END;
END emp_pkg;
/
Package state: Package variables persist for the entire session (not transaction). They are initialized once when the package is first loaded in a session.
QWhat are the advantages of packages over standalone procedures/functions?
APackages provide: (1) Encapsulation — private elements hidden from callers. (2) Performance — the entire package is loaded into the shared pool on first call, making subsequent calls faster. (3) Overloading — multiple subprograms can share a name with different signatures. (4) Session-level state via package variables. (5) Reduced dependency tracking — changing the body without changing the spec does not invalidate dependent objects.
Bulk Collect & FORALL
Reducing context switches, performance
▼

Every switch between the PL/SQL engine and SQL engine is a context switch — expensive at scale. BULK COLLECT fetches multiple rows at once; FORALL issues DML for an entire collection in one SQL engine call.

DECLARE
  TYPE t_ids   IS TABLE OF employees.employee_id%TYPE;
  TYPE t_sals  IS TABLE OF employees.salary%TYPE;
  v_ids   t_ids;
  v_sals  t_sals;
BEGIN
  -- Fetch 500 rows at a time (LIMIT avoids memory overload)
  SELECT employee_id, salary
  BULK COLLECT INTO v_ids, v_sals
  FROM   employees
  WHERE  department_id = 50
  LIMIT  500;

  -- Update all in one SQL call
  FORALL i IN v_ids.FIRST..v_ids.LAST
    UPDATE employees
    SET    salary = v_sals(i) * 1.05
    WHERE  employee_id = v_ids(i);

  DBMS_OUTPUT.PUT_LINE(SQL%BULK_ROWCOUNT.COUNT || ' updated');
END;
/
Rule of thumb: If you're looping over 1000+ rows with row-by-row DML, FORALL can be 10–100x faster.
QWhat is the SAVE EXCEPTIONS clause in FORALL and why is it important?
ABy default, a FORALL statement stops at the first error and rolls back all changes in that batch. SAVE EXCEPTIONS continues processing all rows and collects errors in the SQL%BULK_EXCEPTIONS collection. You can then inspect each error (index, error code) and decide how to handle them. This is critical for bulk loads where you want to commit successful rows and report/reprocess failed ones without aborting the entire batch.
Dynamic SQL — EXECUTE IMMEDIATE & DBMS_SQL
NDS, bind variables, DDL in PL/SQL
▼ 
-- Execute DDL (cannot do this in static PL/SQL)
BEGIN
  EXECUTE IMMEDIATE 'CREATE TABLE temp_log(msg VARCHAR2(200))';
END;
/

-- Dynamic query with bind variable (safe — no SQL injection)
DECLARE
  v_table VARCHAR2(30) := 'EMPLOYEES';
  v_dept  NUMBER       := 10;
  v_cnt   NUMBER;
BEGIN
  EXECUTE IMMEDIATE
    'SELECT COUNT(*) FROM ' || v_table ||
    ' WHERE department_id = :1'
  INTO v_cnt
  USING v_dept;             -- bind variable
  DBMS_OUTPUT.PUT_LINE(v_cnt);
END;
/
SQL Injection risk: Never concatenate user-supplied values directly into dynamic SQL strings. Always use bind variables (USING clause) for data values. Table/column names cannot be bound — validate them against a whitelist.
QWhen would you use DBMS_SQL instead of EXECUTE IMMEDIATE?
AEXECUTE IMMEDIATE (Native Dynamic SQL) handles most cases and is simpler. Use DBMS_SQL when: (1) the number of columns or bind variables is not known at compile time (fully dynamic structure), (2) you need to describe columns after parsing, (3) you need to reuse a parsed cursor for multiple executions with different binds, or (4) you're fetching into dynamically-sized collections. DBMS_SQL is more verbose but more flexible for truly unknown query shapes.
Collections — Nested Tables, VARRAYs, Associative Arrays
INDEX BY, EXTEND, DELETE
▼ 
-- Associative Array (INDEX BY) — best for in-memory lookups
DECLARE
  TYPE t_lookup IS TABLE OF VARCHAR2(100) INDEX BY PLS_INTEGER;
  v_map t_lookup;
BEGIN
  v_map(1) := 'Alice';
  v_map(2) := 'Bob';
  IF v_map.EXISTS(1) THEN
    DBMS_OUTPUT.PUT_LINE(v_map(1));
  END IF;
END;
/

-- Nested Table — can be stored in DB columns
DECLARE
  TYPE t_names IS TABLE OF VARCHAR2(50);
  v_names t_names := t_names('Alice', 'Bob');
BEGIN
  v_names.EXTEND;            -- add one slot
  v_names(3) := 'Carol';
  v_names.DELETE(2);         -- delete index 2
  DBMS_OUTPUT.PUT_LINE(v_names.COUNT);  -- 2
END;
/
QWhat is the difference between a nested table and a VARRAY?
AA VARRAY (variable-size array) has a fixed maximum size declared at definition time, elements are always dense (no gaps), and when stored in the database they are stored inline with the row. A nested table has no maximum size, can become sparse after deletions, and when stored in the database is stored in a separate storage table. Use VARRAYs for small, fixed-bound ordered collections; use nested tables for larger or unbounded collections that need set operations.
Autonomous Transactions
PRAGMA AUTONOMOUS_TRANSACTION, audit logging
▼

An autonomous transaction runs independently of its parent transaction. COMMITs and ROLLBACKs inside it do not affect the caller's transaction.

CREATE OR REPLACE PROCEDURE log_error(
  p_msg IN VARCHAR2
) AS
  PRAGMA AUTONOMOUS_TRANSACTION;
BEGIN
  INSERT INTO error_log(msg, logged_at)
  VALUES (p_msg, SYSDATE);
  COMMIT;  -- commits only the log insert
END;
/

-- Caller rolls back its own work but log is preserved
BEGIN
  INSERT INTO orders ...; -- business insert
  log_error('Something went wrong');
  ROLLBACK;   -- order insert rolled back; log stays
END;
Classic use case: Audit/error logging that must persist even when the main transaction rolls back.
QWhat are the risks of overusing autonomous transactions?
AAutonomous transactions can cause data inconsistency — the autonomous transaction commits data that the parent later rolls back, leaving orphaned or inconsistent rows. They also consume an extra transaction slot and can cause deadlocks if the autonomous transaction tries to lock a row that the parent already holds. Use them only for logging, auditing, or other side-effects that should be independent of the parent's outcome — never as a shortcut to commit mid-procedure.
